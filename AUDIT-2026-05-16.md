# TMW OS Audit — 2026-05-16

Full top-to-bottom audit of the tm-weddings-tracker repo at HEAD
(`537d420` — "wallpaper(w9): Julia & Ethan timeline lock screen").

12,322 lines of HTML/JS across 24 files. 12 SQL paper-trail entries.
Two shared foundation scripts. One untracked workflow doc.

This audit is opinionated. It calls out the things that will bite you,
ranked by how soon they will bite. Architectural patterns that already
work fine are not relitigated.

---

## TL;DR — The five things that matter

1. **`auth/callback.html` has a parallel hardcoded `APPROVED_EMAILS` list.**
   This is the actual gate at login. If a new team member is added to
   `tmw_team_members` in Supabase but not to this file, they cannot log
   in. The Greg + Nick onboarding both required hand-edits to this file
   (commits `911923c`, `d6f28e5`). This is your single highest-leverage
   fix.

2. **The admin password `Post-Production_2026!` is in plain text in
   `tmw-core.js`.** Anyone who opens DevTools sees it. It gates "admin
   mode" in 5 pages. The owner check via `TMW_USER.isOwner` is the only
   real security; the password just lets non-owner editors pretend.
   Either delete the password gate or move it to a real role check.

3. **CLAUDE.md is out of sync with the actual repo.** It says `tmw_state`
   is "misc app state" — it is actually the video projects store. It
   doesn't mention the `marketing/` folder (1,856 lines), `referrals/program.html`,
   or `post-production/content/`. The `tmwIsAdmin()` signature documented
   is wrong. Fix the doc or future Claude sessions will keep making the
   same wrong assumptions.

4. **`OWNER_EMAILS` is now hardcoded in 9 places, not the 3 CLAUDE.md
   admits to.** Plus a separate `EDITOR_EMAILS` map duplicated three
   different ways. Bucket 2.5 ("Consolidate OWNER_EMAILS") needs to be
   pulled forward.

5. **Every dashboard polls every 30 seconds and writes the entire
   `projects.data` array on every edit.** Concurrent edits silently
   overwrite each other. Bucket 2.6 should add row-level optimistic
   concurrency or move to per-entity rows.

If you do nothing else this month, do these five. Everything below is
detail and lower priority.

## 1. Architecture: what's working

Don't change these. They earned their keep.

- **Centralized `tmwSbFetch` with JWT injection.** Cleanly enforced
  across every feature file. The refresh-token logic added 2026-05-06
  is solid and the right call.
- **Folder-per-feature layout.** Every new feature lives in its own
  top-level dir. Stays predictable as the surface grows.
- **`tmw-auth.js` declarative gate.** `<script data-required-role="editor">`
  is a clean pattern. Easy to audit who can see what just by grepping.
- **External role path-restriction.** The marketing-only role being
  redirected back to `/marketing/*` is a smart, future-proof primitive
  for client-facing or partner-facing surfaces later.
- **SQL paper trail in `/sql-history/`.** Every schema change has a
  dated file with rationale. This will save you when something breaks
  in production six months from now.
- **WEDDING_INTAKE.md.** Best piece of writing in the repo. It exists
  because intake kept failing. Commit it (currently untracked).

## 2. Critical issues (fix this month)

### 2.1 Auth allowlist drift

**File:** `auth/callback.html` lines 50-67

There is a hardcoded `APPROVED_EMAILS` array of 16 emails. This runs at
OAuth callback time, BEFORE `tmw-auth.js` ever sees the user. If an email
isn't here, login fails with "Email not authorized" regardless of what's
in `tmw_team_members`.

**Why this is bad:**
- Onboarding now requires editing this file AND Supabase, plus a deploy.
- Offboarding requires removing the email here OR they keep logging in
  (until they hit the second-stage check in `tmw-auth.js`).
- Tony adds people in `team-members.html`. The page lies about what it
  takes to add a teammate.

**The fix:**

In `auth/callback.html`, replace the static `APPROVED_EMAILS` array with
a synchronous fetch to `projects?id=eq.tmw_team_members&select=data` using
the new JWT we just got from Supabase. Pseudocode:

```js
var jwt = decodeJWT(tokens.access_token);
var email = jwt.email.toLowerCase();

// Fetch the live team list using the JWT we just received
var xhr = new XMLHttpRequest();
xhr.open('GET', SUPABASE_URL + '/rest/v1/projects?id=eq.tmw_team_members&select=data', false);
xhr.setRequestHeader('apikey', SUPABASE_KEY);
xhr.setRequestHeader('Authorization', 'Bearer ' + tokens.access_token);
xhr.send(null);
var rows = JSON.parse(xhr.responseText);
var approved = (rows[0].data.members || [])
  .filter(function(m){ return m.active !== false; })
  .map(function(m){ return m.email.toLowerCase(); });

if (approved.indexOf(email) === -1) { /* show error */ }
```

Keep a 2-email emergency fallback (Tony + Brynn) for the case where the
fetch fails, mirroring `EMERGENCY_OWNERS` in `tmw-auth.js`.

After this lands: onboarding is one row in `team-members.html` and zero
file edits.

### 2.2 Admin password in client-side JS

**File:** `tmw-core.js` line 24 → `password: 'Post-Production_2026!'`

Read in `weddings/index.html`, `weddings/client.html`, `post-production/photo/index.html`,
`post-production/deliverables/index.html`, `post-production/video/index.html`.

**The actual security model right now:**
- Owners (Tony, Brynn) bypass the password via `TMW_USER.isOwner` check.
- Editors who know the password can self-promote to "admin" in any
  dashboard and edit project structure.
- The password is visible to anyone in DevTools.

**Pick one of these:**

**Option A (recommended): delete the password entirely.** Make admin
features owner-gated only. If non-owner editors need to edit deliverables
or due dates, give them an `editor` role flag in `tmw_team_members` and
gate the UI on `TMW_USER.isAdmin || TMW_USER.isEditor`.

**Option B: move the check server-side.** RLS policy on `projects` that
requires `auth.email() IN (...)` for UPDATE. Then the password becomes
optional UX, not security.

The current Option-C of "password in client code" provides no security
and just adds friction.

### 2.5 Lost-update problem on every save

**Pattern:** Every post-production page does `PATCH projects?id=eq.tmw_photo
{data: {projects: [...entire array...]}}` on every single field change.

If Tony edits a stage on `p1d2` (Casey Gruver gallery) and Brynn edits a
stage on `p5d1` (different client) at the same time, the second save
overwrites the first because both pulled the same baseline array, mutated
their own slice, and wrote the whole thing back.

This has probably already cost you data without you knowing — silent
loss, no error. The 30s polling masks the symptom by re-syncing on the
next interval, but anything edited in that window is gone.

**The fix (cheap version):** add an `If-Match` header using a `version`
or `updated_at` timestamp. PostgREST supports this. If the server's
`updated_at` doesn't match what the client thinks it is, the save fails
with 412 and the page reloads.

**The fix (better):** split the projects array into individual rows
keyed by `(type, project_id)` and update one row at a time. This is the
real Bucket 8 ("Proper team_members Relational Table") generalized to
projects. Multi-week project; do it when post-prod volume actually
demands it.

### 3.2 Two auth idioms still coexist

CLAUDE.md rule #3 acknowledges this and asks new code to "match the
pattern of the surrounding file." Reality:

- Pattern A (`TMW_USER.isOwner`) is used in `index.html`, `weddings/index.html`,
  `post-production/photo/index.html`, `shot-caller/index.html`,
  `dashboard/build-calendar/index.html`.
- Pattern B (`tmwIsAdmin()`, `tmwHasRole()`) is exposed by `tmw-auth.js`
  but barely used anywhere.
- `tmw-core.js` defines a SEPARATELY-NAMED `tmwIsAdmin(sessionKey)` that
  reads sessionStorage for the "admin mode unlocked" flag. **The
  `tmwIsAdmin` in `tmw-core.js` and the `tmwHasRole` in `tmw-auth.js`
  do completely different things despite identical naming patterns.**

This is worse than CLAUDE.md admits. It's not two idioms, it's two
idioms PLUS a name collision.

**The fix:** Pick Pattern A. Delete `tmwHasRole` and `tmwCanAccess` from
`tmw-auth.js` (they're unused in the rest of the codebase per my grep).
Rename the `tmwIsAdmin` in `tmw-core.js` to `tmwIsAdminUnlocked` or
`tmwAdminModeOn`. One sweep.

### 3.3 30-second polling on six dashboards

`setInterval(loadData, 30000)` in:
- `post-production/photo/index.html`
- `post-production/deliverables/index.html`
- `post-production/video/index.html`
- `post-production/content/index.html`
- `weddings/index.html`
- `dashboard/index.html`

Math: 6 dashboards × ~15 active team members × 2,880 calls/day = ~260K
Supabase reads per day if everyone leaves a tab open. Supabase free tier
is generous but the auth refresh request flood matters too.

More importantly: 30s polling is the wrong primitive. It's slow when you
want immediate feedback (someone marks a stage complete, you wait up to
30s to see it) and wasteful when nothing is happening.

**The fix:** Switch to Supabase Realtime subscriptions on the rows that
change frequently (`tmw_photo`, `tmw_state`/`tmw_video`, `tmw_weddings`).
This is a 1-day swap. Keep the manual reload button.

## 4. Medium-priority issues (next quarter)

### 4.1 RLS uniformly permissive

Every policy is `{authenticated}` with `USING (true)`. All access control
is in JS. Anyone with a valid JWT can read or write any row.

CLAUDE.md acknowledges this and queues Bucket 2.5 for owner-level RLS.
Two things to add to that bucket:

- **Editor write isolation:** an editor should only be able to mutate
  rows that involve them (e.g., a deliverable assigned to them). RLS
  on `tmw_photo` checking `data.deliverables[].editor = auth.jwt() ->> 'email'`
  is doable via `WHERE` clause.
- **Referrals visibility:** today every authenticated user can read every
  referral (including payout amounts). At minimum, non-owners should
  only see referrals where `referrer_id = auth.email()`.

### 4.2 Notification routing is broken for most editors

**Worse than it looks.** `tmwNotify` in `tmw-core.js` (line 85-113) is
the centralized notification helper. It is **called zero times in the
entire codebase**. Every post-prod page calls `emailjs.send` directly
with its own (inconsistent) hardcoded recipient logic, and only Tony +
Brynn (sometimes + Lexi) actually get emails.

**The fix:** Both `tmwNotify` and `tmwNotifyAssignment` should resolve
recipients from `tmw_team_members` at call time, not from a hardcoded
map. Then sweep every page that calls `emailjs.send` directly and
replace with `tmwNotify(...)`. Once 2.4 is done this falls out for free.

### 4.3 EmailJS template reuse

`template_p1hyj87` was a post-production stage template; referrals reuses
it with field-mapping hacks. CLAUDE.md notes this; the bucket "Referrals-specific
EmailJS Template" is queued. Ship it — confusing emails to clients/referrers
will eventually cost you a booking.

### 4.4 Schema migrations are manual + tribal

`/sql-history/` is a good paper trail but it's not idempotent and not
replayable. Every schema change goes through the Supabase dashboard SQL
editor. If you ever stand up a staging environment, you have nothing.

CLAUDE.md lists "proper migrations" as a Bucket 3 item. When you get
there, look at supabase CLI's migration support — `supabase db diff`
and `supabase migration new` are good primitives. The transition would
be: continue writing SQL files in `/sql-history/`, but also commit them
as `supabase/migrations/<timestamp>_<name>.sql`.

### 4.5 No error monitoring

Errors are `console.warn`'d. Users see toasts at best. You will never
know about an EmailJS quota cap, a Supabase 5xx, or a JS exception in
the wild unless someone tells you.

Sentry is queued in the build plan. Move it up. It's 30 lines of
boilerplate and pays for itself the first time something silently breaks.

### 4.6 `tmwNotifyAssignment` redefined inline

`post-production/video/index.html` lines 20-31 redefine `tmwNotifyAssignment`
*after* loading `tmw-core.js` (which already exports its own version on
line 116-135). The video version uses a different `EDITOR_EMAILS` map.

Pick one. Probably the core version (after 4.2's fix makes it dynamic).

## 5. Low-priority observations

These are not urgent but worth knowing.

- **Inline event handlers everywhere** (`onclick="..."`). Means you can't
  add a Content Security Policy without `'unsafe-inline'`, which defeats
  most of CSP's value. Not a problem until it is.
- **No tests anywhere.** Zero test files. Manual smoke testing only.
  Acceptable for the current scale; risky once a second engineer touches
  the code.
- **No CSP, no SRI** on any CDN script. EmailJS + Supabase + Google Fonts
  all loaded from third parties without integrity hashes.
- **Cosmetic CSS variable drift:** `--accent` color is `#8b5e4a` (post-prod
  rose) on some pages and `--teal` `#4aacac` on others. Two distinct
  visual systems coexist. Probably intentional but worth deciding which
  is canonical.
- **`tm-reels/` confusion:** lives in this repo but appears unrelated to
  TMW OS. Either link to it from the portal or extract.
- **Refresh tokens in localStorage:** XSS-readable. Amplified by every
  page running 200-700 lines of inline JS. Single XSS bug → 7-day token
  theft. Mitigations: CSP + SRI + httpOnly cookie alternative are all
  hard pivots; for now, just be aware.

These are about how YOU work the system, not the code.

### 6.1 Bucket priorities are misordered

**Concrete reorder:**

   the same problem from two angles).
2. **3.2 Stage label dedup** (small, follow-on to 2.6).

### 6.2 Wedding intake workflow has friction

WEDDING_INTAKE.md is well-designed. The friction point is steps 4-5:
manual SQL paste + manual wallpaper render. Tony pastes intake, COO
returns SQL, Tony pastes SQL into Supabase, then renders wallpaper, then
commits wallpaper, then pastes another UPDATE for `wallpaperUrl`.

**The single highest-leverage win:** add a `/dashboard/wedding-intake.html`
admin page that accepts the freeform intake text, POSTs to a Supabase
edge function that parses it (LLM call) and writes the row directly. Two
steps collapse to one. Wallpaper render stays manual until someone
automates Playwright in a worker.

This is already listed as "the bulk freeform intake page is a queued
roadmap item." Move it up.

### 6.3 SQL paper trail headers are inconsistent

Header formats across `/sql-history/`:

- Some use `===...===` borders, some use box-drawing chars.
- Some have `Bucket:` field, some don't.
- Some have explicit `RLS notes:` and `Rollback notes:`, some skip.

Five-line template would solve it. Add to CLAUDE.md or `WEDDING_INTAKE.md`:

```sql
-- ────────────────────────────────────────────────────────
-- File:    YYYY-MM-DD-<description>.sql
-- Bucket:  <bucket_id or feature name>
-- Author:  Tony Martinez
-- Date:    YYYY-MM-DD
-- Purpose: <one sentence>
-- RLS:     <changes or "no changes">
-- Rollback: <how to undo>
-- ────────────────────────────────────────────────────────
```

### 6.4 No deployment checklist

Every commit auto-deploys to Vercel. No staging environment. No
pre-deploy smoke test. The Greg email-alias bug (`d6f28e5` followed by
`289cf7a` one commit later) was visible only after deploy.

Even a single-page `/dashboard/smoke-test.html` that you visit after a
deploy and which exercises (a) auth, (b) read from `tmw_team_members`,
(c) read from `tmw_photo`, (d) write a no-op PATCH and revert — would
have caught that.
